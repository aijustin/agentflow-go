agentflow-go 使用手册

把 Agent 能力嵌入你的 Go 服务

AgentFlow Go 是面向 Go 后端工程师的可嵌入 Agent 运行时库：用 pkg/builder 或 core.Scenario 定义场景，Go 代码注入模型、持久化、治理与 HTTP Handler。不提供官方 CLI 或独立 Server 二进制。

开始接入 开启可观测 查看企业能力 阅读 README 示例路径
发布就绪路径 
go get github.com/aijustin/agentflow-go

go run ./examples/go/builder/main.go
go run ./examples/go/validate -kind builder all
make validate-builder

GOTOOLCHAIN=auto make release-check

03 / 心智模型

核心概念

智能体

业务角色。绑定 LLM profile、工具列表、系统指令、记忆和输出 schema。

工具

可执行能力。通过 `core.ToolExecutor` 接入，经过 allowlist、RBAC、审批、速率限制、审计和输出脱敏。

LLM Profile

模型配置。声明 provider、model、context window、能力矩阵、timeout、temperature 和 extra body。

记忆

会话或命名空间记忆。运行前注入上下文，运行后写入用户、助手和工具观察。

运行状态

可恢复执行快照。用于 HITL 暂停、workflow 节点状态、CAS 版本和跨进程恢复。

工作流

固定图编排。支持依赖、条件表达式、重试、set/copy 转换节点、Agent 节点、人工门禁和并发批次。

技能包

声明式 prompt fragment、Agent policy、Tool policy 和 workflow segment，会在场景构建阶段展开。

可观测事件

Runtime 通过 EventSink 输出 Run、Step、Tool、LLM、Memory、Context 和 HITL 事件，可持久化并实时推送。

身份与治理

Principal、RBAC、API Key、OIDC/JWKS JWT、工具授权、预算、side effect 和输出脱敏统一走策略端口。

04 / 场景构造

Go Builder 场景

场景在 Go 代码中构造，类型安全、可重构、可测试。推荐从 preset stack 或 ScenarioBuilder 开始；复杂 workflow 用 WorkflowBuilder，命名子图用 NamedWorkflow，动态 fan-out 用 NodeMap。

校验: agentflow.ValidateScenario、make validate-builder、go run ./examples/go/validate -kind builder all。完整 API 见 Builder 速查 与 配置参考（描述 core.Scenario 字段）。
main.go
import (
  agentflow "github.com/aijustin/agentflow-go"
  "github.com/aijustin/agentflow-go/pkg/builder"
)

scenario := builder.MinimalAutonomous("assistant")
prep := builder.NewWorkflow().
  NodeTransform("mark", json.RawMessage(`{"set":{"ready":true}}`)).
  Build()
mainWF := builder.NewWorkflow().
  NodeSubgraph("run_prep", "prep").
  Build()
advanced := builder.New("advanced").
  DefaultMockLLM().
  Agent("assistant").DefaultLLM().Done().
  NamedWorkflow("prep", prep).
  FixedWorkflow(mainWF).
  Scenario()

fw, err := agentflow.New(advanced, opts...)

07 / 工具体系

工具体系

所有工具都作为 core.ToolExecutor 接入。内置工具保持收紧默认值，企业自定义工具也应遵循同样模型: 输入 schema、side effect、approval、rate cap、审计和输出脱敏。工具声明和执行器绑定是分离的：scenario.tools 暴露 manifest，WithToolExecutor 适合轻量常驻工具，WithToolResolver 适合在通过 allowlist、审批、RBAC 和治理策略后再按需解析重型或租户隔离工具。

HTTP 工具

读取 allowlisted 内部 API。默认仅允许 GET/HEAD，限制 host、method 和响应大小。

builtin.http

文件系统工具

读取 allowlisted root 下的本地文件。拒绝路径逃逸、符号链接逃逸和超大文件。

builtin.filesystem

SQL 工具

执行命名 read-only SQL。支持 PostgreSQL、MySQL、ClickHouse 的 `database/sql` 驱动。

builtin.sql

MCP 工具

把 HTTP 或 stdio MCP Server 包装成普通 ToolExecutor，继续使用 allowlist、审批、速率限制和审计。

mcp

Retriever 工具

组合 Embedder、VectorStore 和 namespace/filter，把 pgvector 检索结果带 citation 返回给 Agent。

rag

子 Agent

supervisor 可以把 sub_agents 当作虚拟 delegation tool 调用，复用同一套事件和治理模型。

delegate
SQL 工具装配
sqlTool, err := agentflow.NewSQLToolExecutor(agentflow.SQLToolConfig{
  DB: db,
  AllowedQueries: map[string]string{
    "tickets.open": "SELECT id, title, status FROM tickets WHERE status = $1",
  },
  MaxRows: 20,
})

fw, err := agentflow.New(
  builder.MinimalSQLTool("assistant"),
  agentflow.WithToolExecutor("sql.query", sqlTool),
)
惰性工具解析
resolver := agentflow.ToolResolverFunc(func(ctx context.Context, tool core.Tool) (core.ToolExecutor, error) {
  switch tool.Type {
  case "builtin.sql":
    return newTenantSQLTool(ctx, tool.Metadata)
  case "mcp.tool":
    return newTenantMCPTool(ctx, tool.Metadata)
  default:
    return nil, fmt.Errorf("unsupported tool type %q", tool.Type)
  }
})

scenario := builder.MinimalAutonomous("assistant")
fw, err := agentflow.New(
  scenario,
  agentflow.WithToolResolver(resolver),
)
Skill 边界: Skill 解决 prompt、policy 和 workflow 片段的复用，会在场景构建阶段展开；它不初始化工具。真实工具客户端、租户凭证和重型连接器应由 WithToolResolver 在调用时绑定。
08 / 知识库

RAG 与知识库

索引链路

  1. Loader 从文件或 HTTP 拉取文档。
  2. Chunker 切分文本并保留 parent_id、chunk_index、位置元数据。
  3. Embedder 生成向量。
  4. VectorStore 写入 pgvector 或自定义实现。

检索链路

  1. Retriever tool 接收 query。
  2. Embedder 生成 query vector。
  3. VectorStore 执行 namespace 和 metadata filter。
  4. 结果返回 content、score、metadata、citation。
检索工具装配
store, err := agentflow.NewPostgresVectorStore(agentflow.PostgresVectorStoreConfig{DB: db})
retriever, err := agentflow.NewRetrieverTool(agentflow.RetrieverToolConfig{
  Embedder:     provider,
  Store:        store,
  Profile:      "embed",
  Namespace:    "tenant-a/docs",
  DefaultLimit: 5,
})

10 / 事件与异步

事件路由与异步 API

Webhook + Queue

复杂现场场景通常需要两类入口：外部系统通过 Webhook 触发 Agent，以及长任务通过队列异步执行。AgentFlow 用 scenario.triggers 描述事件映射，用 NewProductionHTTPHandler 统一挂载健康检查、异步 job API 和可选的同步 event/HITL 路由。

13 / 落地采用

落地检查表

开发阶段

  • 每个工具都有输入 schema、side effect 和 approval 策略。
  • LLM profile 显式声明 capabilities。
  • 需要多模型时使用 Provider Router，chat/tool/structured/stream/embed 按 profile 路由。
  • 复杂自主任务开启 planning pass，并把计划写入后续上下文。
  • 需要恢复的流程使用持久化 RunStateRepository。
  • RAG 结果带 source metadata 和 citation，VectorStore 按 namespace 隔离。
  • Webhook/工单场景为 scenario.triggers 定义 event、agent 和 JSON 路径映射。
  • 用 ValidateWiring 在 New 前校验场景；示例可用 examples/go/validate。
  • 生产环境显式 WithLLMGateway 与 WithToolExecutor；测试可用 pkg/testutil.WiringOptions。
  • HTTP + Worker 参考 examples/go/http-worker，挂载 NewProductionHTTPHandler 与 NewFrameworkJobHandler。
  • 需要削峰时走 /v1/jobs/events 和 /v1/jobs/hitl/resume。
  • Git/ticket 等工具须显式注册执行器，详见 tools-git.md 与 tools-ticket.md。
  • 审批后续跑使用 ResumeAndContinue（HTTP continue: true 或示例 hitl-resume）。
  • 事件 payload 只放运营元数据，敏感输出先走 redactor。
  • 新增公共 API 同步 README、CHANGELOG 和 godoc。

生产阶段

  • 宿主 HTTP 服务配置 HITL token secret、API key 或 JWT middleware。
  • 可观测面板挂载在受保护路径，并启用 PostgreSQL EventStore。
  • 受控数据库环境先跑 migration，再对 EventStore 设置 `SkipSchemaSetup`。
  • 企业认证优先接入 OIDC/JWKS JWT，并保留 tenant/workspace/project scope。
  • 数据库工具使用只读账号和最小表权限。
  • 审计日志覆盖审批、拒绝、修订和高风险工具调用。
  • 对象存储、数据库、Redis 由平台统一管理密钥。
  • 发版前运行 `GOTOOLCHAIN=auto make release-check`。
